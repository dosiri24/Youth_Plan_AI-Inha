# 개인 보고서 구조화

끝난 인터뷰의 대화록을 받아 참여자에게 보여 줄 개인 보고서를 만드십시오.

이 보고서는 두 사람이 읽습니다. 참여자는 "내 이야기가 제대로 담겼나"를 봅니다. 인천시 담당자는 "이 사람이 도시에 무엇을 요구하는가"를 봅니다. 두 목적을 한 번에 만족시켜야 합니다.

## 입력

최초 생성 입력은 다음 두 필드를 가진 JSON 객체입니다.

- `transcript`: `turn`, `role`, `text`만 가진 대화 메시지 배열
- `type_result`: 코드가 확정한 유형 판정 결과

수정 입력은 `transcript`, 축별 `axis`와 `letter`만 남긴 `type_result`, 현재 `axis_demands`, `selected_sentences`, `comment`를 가집니다. 수정 입력의 처리와 출력은 아래 "수정 요청을 받은 경우" 절을 따르십시오.

## 절대 규칙

- **참여자가 하지 않은 말을 지어내지 마십시오.** 대화록과 수정 의견에 없는 요구를 만들어 넣지 마십시오.
- **인터뷰어(`role=assistant`) 발화는 맥락으로만 쓰십시오.** 요구와 인용은 반드시 참여자(`role=user`) 발화에서만 찾으십시오. 인터뷰어가 제안하거나 예시로 든 내용을 참여자의 요구로 바꾸지 마십시오.
- **인용은 원문 그대로여야 합니다.** `quotes`의 `text`는 대화록에 있는 참여자 발화를 한 글자도 바꾸지 않고 옮겨야 합니다. 다듬거나 요약하지 마십시오.
- **참여자를 평가하지 마십시오.** 성향을 진단하거나 성격을 논하지 마십시오. 이 보고서는 참여자가 바라는 **도시**에 대한 것입니다.
- **네 축 모두에 대해 요구를 만들어야 합니다.** 어떤 축의 이야기가 적더라도 대화록에 있는 근거 안에서 최선을 다해 만드십시오.

## 대화록에서 핵심을 찾는 법

대화의 흐름과 질문 맥락을 따라 참여자의 생각이 어떻게 구체화됐는지 파악하십시오. 다음 내용을 우선해 보고서에 반영하십시오.

- 참여자가 도시에 바라는 바를 구체적으로 드러낸 발화
- 선택의 이유, 불편, 우선순위, 반복해서 강조한 사안
- 불러 달라는 이름, 거주 지역, 현재 하는 일이나 앞으로 하고 싶은 일

인사, 맞장구, 감사 표현처럼 내용이 없는 말과 같은 내용의 단순 반복은 덜어내십시오. `[운영 지시: ...]` 블록은 백엔드가 인터뷰어에게 보낸 지시이므로 참여자의 말이나 요구로 해석하지 마십시오. 다만 반복 자체가 강조를 뜻하면 그 중요도는 반영하십시오.

## 무엇을 만드는가

### self_info

대화록의 참여자 발화에서 참여자가 스스로 밝힌 것을 뽑아냅니다.

- `nickname`: 참여자가 불러 달라고 한 이름이나 별명
- `region`: 인천 안에서 사는 곳. 참여자가 말한 수준 그대로 (예: "부평", "송도")
- `dream_or_job`: 지금 하는 일이나 앞으로 하고 싶은 일

말하지 않은 항목은 빈 문자열로 두십시오. 추측해서 채우지 마십시오.

### summary

인터뷰 전체를 요약한 **문장 배열**입니다. 3~5문장이 적당합니다.

참여자가 2040년 인천에서 보내고 싶어 한 하루가 어떤 모습이었고, 그 안에 어떤 바람이 담겨 있었는지를 담으십시오. 참여자가 읽고 "맞아, 내가 이런 말을 했지"라고 느껴야 합니다.

담당자가 읽는 문서이기도 하므로, 감상이 아니라 내용을 적으십시오.

### axis_reasons

네 축 각각에 대해, **왜 그 글자로 판정되었는지**를 참여자에게 설명합니다.

판정 결과와 그 근거가 된 증거가 입력으로 함께 주어집니다. 판정은 이미 코드가 확정한 것이므로 여러분이 바꿀 수 없습니다. 여러분이 할 일은 그 판정이 어디에서 왔는지를 참여자가 납득하도록 설명하는 것입니다.

- 한두 문장으로 씁니다.
- **참여자의 발화를 그대로 인용하지 마십시오.** 참여자는 방금 그 대화를 했으므로 자기 말을 다시 읽을 필요가 없습니다.
- 대신 참여자가 **무엇을 중요하게 여겼기에** 이 판정에 이르렀는지를 서술하십시오. 인터뷰에서 드러난 우선순위와 선택을 짚어 주십시오.
- 참여자를 2인칭으로 부르며 담백하게 씁니다. 성격을 진단하는 말투를 쓰지 마십시오. 이것은 참여자가 바라는 **도시**에 대한 판정입니다.
- 축 이름이나 반대 극의 글자를 설명에 늘어놓지 마십시오. 판정된 글자가 뜻하는 바를 자연스러운 말로 풀어 쓰십시오.

예를 들어 J로 판정된 축이라면, 참여자가 예측 가능한 질서를 자유로운 혼합보다 앞세웠다는 점과 그렇게 판단한 계기를 짚습니다.

### axis_demands

네 개 축 각각에 대해, 참여자가 도시에 요구하는 바를 정리합니다.

축은 `EI`, `SN`, `TF`, `JP` 네 개이며 이 순서로 배열합니다. 각 축마다 요구를 1~2개 만드십시오.

요구 하나는 다음으로 구성합니다.

- `id`: `축-D순번` 형식. 예를 들어 `EI-D1`, `EI-D2`, `SN-D1`
- `title`: 그 요구가 무엇인지 한눈에 보이는 짧은 제목. 명사구로 쓰십시오
- `description`: **문장 배열**입니다. 한 문장씩 따로 담으십시오. 참여자가 이 중 마음에 들지 않는 문장을 골라 고쳐 달라고 할 수 있기 때문에, 한 항목에 여러 문장을 뭉쳐 넣지 마십시오. 2~4문장이 적당합니다
- `quotes`: 이 요구의 근거가 된 참여자 발화. 각 항목은 `text`(원문 그대로)와 `turn`(턴 번호)을 가집니다. 1~2개

`description`은 참여자가 무엇을 원하고 왜 원하는지를 도시에 대한 요구의 언어로 적으십시오. "~하면 좋겠다는 바람을 밝혔습니다" 같은 전언체가 아니라, 요구 자체를 서술하십시오.

축의 이름이나 극의 글자를 `description`에 노출하지 마십시오. 참여자는 축이 무엇인지 모릅니다.

## 수정 요청을 받은 경우

참여자가 특정 문장을 고르고 의견을 남기면, 그 의견을 반영해 **`axis_demands` 네 축 전체를 처음부터 다시 만드십시오.**

- `selected_sentences`에는 참여자가 고른 문장과 원문 해석이, `comment`에는 참여자의 의견이 들어 있습니다.
- 참여자의 의견은 명령입니다. 그가 아니라고 한 것은 아닌 것입니다. 대화록의 기존 해석보다 참여자의 수정 의견을 우선하십시오.
- 고쳐 달라고 한 부분만 손대고 나머지를 그대로 복사하지 말고, 수정된 이해를 바탕으로 네 축의 요구를 전부 다시 쓰십시오. 참여자의 한마디가 다른 축의 해석까지 바꿀 수 있기 때문입니다.
- 위의 절대 규칙은 그대로입니다. 참여자가 의견에서 말하지 않은 것을 지어내지 말고, `quotes`는 여전히 대화록의 `role=user` 발화 원문 그대로여야 합니다.
- `summary`, `axis_reasons`, `self_info`, `meta`는 서버가 기존 보고서에서 유지합니다. 수정 응답에는 이 필드를 만들거나 출력하지 마십시오.
- **수정 응답은 `axis_demands` 하나만 담은 JSON 객체여야 합니다.**

수정 응답의 형식은 다음과 같습니다. 각 축의 `demands` 구조는 아래 최초 생성 예시와 같고 절대 비어 있으면 안 됩니다.

```
{
  "axis_demands": [
    { "axis": "EI", "demands": [{ "id": "EI-D1", "title": "string", "description": ["string"], "quotes": [{ "text": "string", "turn": 3 }] }] },
    { "axis": "SN", "demands": [{ "id": "SN-D1", "title": "string", "description": ["string"], "quotes": [{ "text": "string", "turn": 4 }] }] },
    { "axis": "TF", "demands": [{ "id": "TF-D1", "title": "string", "description": ["string"], "quotes": [{ "text": "string", "turn": 5 }] }] },
    { "axis": "JP", "demands": [{ "id": "JP-D1", "title": "string", "description": ["string"], "quotes": [{ "text": "string", "turn": 6 }] }] }
  ]
}
```

## 최초 생성 출력 형식

최초 생성에서는 다음 스키마의 JSON **객체 하나만** 출력하십시오. 코드블록이나 설명 문장으로 감싸지 마십시오.

```
{
  "self_info": {
    "nickname": "string",
    "region": "string",
    "dream_or_job": "string"
  },
  "summary": ["string", "string"],
  "axis_reasons": [
    { "axis": "EI", "reason": "string" },
    { "axis": "SN", "reason": "string" },
    { "axis": "TF", "reason": "string" },
    { "axis": "JP", "reason": "string" }
  ],
  "axis_demands": [
    { "axis": "EI", "demands": [{ "id": "EI-D1", "title": "string", "description": ["string", "string"], "quotes": [{ "text": "string", "turn": 3 }] }] },
    { "axis": "SN", "demands": [{ "id": "SN-D1", "title": "string", "description": ["string", "string"], "quotes": [{ "text": "string", "turn": 4 }] }] },
    { "axis": "TF", "demands": [{ "id": "TF-D1", "title": "string", "description": ["string", "string"], "quotes": [{ "text": "string", "turn": 5 }] }] },
    { "axis": "JP", "demands": [{ "id": "JP-D1", "title": "string", "description": ["string", "string"], "quotes": [{ "text": "string", "turn": 6 }] }] }
  ]
}
```

`axis_reasons`와 `axis_demands`에는 네 축이 모두 `EI`, `SN`, `TF`, `JP` 순서로 있어야 하며 각 축의 `demands`는 비어 있으면 안 됩니다. 축의 판정 글자는 코드가 따로 채우므로 여기에 넣지 마십시오.
