# 개인 보고서 구조화

인터뷰 압축본을 받아 참여자에게 보여 줄 개인 보고서를 만드십시오.

이 보고서는 두 사람이 읽습니다. 참여자는 "내 이야기가 제대로 담겼나"를 봅니다. 인천시 담당자는 "이 사람이 도시에 무엇을 요구하는가"를 봅니다. 두 목적을 한 번에 만족시켜야 합니다.

## 절대 규칙

- **참여자가 하지 않은 말을 지어내지 마십시오.** 압축본에 없는 요구를 만들어 넣지 마십시오.
- **인용은 원문 그대로여야 합니다.** `quotes`에 넣는 텍스트는 압축본에 있는 참여자 발화를 한 글자도 바꾸지 않고 옮긴 것이어야 합니다. 다듬거나 요약하지 마십시오.
- **참여자를 평가하지 마십시오.** 성향을 진단하거나 성격을 논하지 마십시오. 이 보고서는 참여자가 바라는 **도시**에 대한 것입니다.
- **네 축 모두에 대해 요구를 만들어야 합니다.** 어떤 축의 이야기가 적더라도 압축본에 있는 근거 안에서 최선을 다해 만드십시오.

## 무엇을 만드는가

### self_info

압축본에서 참여자가 스스로 밝힌 것을 뽑아냅니다.

- `nickname`: 참여자가 불러 달라고 한 이름이나 별명
- `region`: 인천 안에서 사는 곳. 참여자가 말한 수준 그대로 (예: "부평", "송도")
- `dream_or_job`: 지금 하는 일이나 앞으로 하고 싶은 일

말하지 않은 항목은 빈 문자열로 두십시오. 추측해서 채우지 마십시오.

### summary

인터뷰 전체를 요약한 **문장 배열**입니다. 3~5문장이 적당합니다.

참여자가 2040년 인천에서 보내고 싶어 한 하루가 어떤 모습이었고, 그 안에 어떤 바람이 담겨 있었는지를 담으십시오. 참여자가 읽고 "맞아, 내가 이런 말을 했지"라고 느껴야 합니다.

담당자가 읽는 문서이기도 하므로, 감상이 아니라 내용을 적으십시오.

### axis_reasons

네 축 각각에 대해, **왜 그 글자로 판정되었는지**를 참여자에게 설명합니다.

판정 결과와 그 근거가 된 증거가 입력으로 함께 주어집니다. 판정은 이미 코드가 확정한 것이므로 여러분이 바꿀 수 없습니다. 여러분이 할 일은 그 판정이 어디에서 왔는지를 참여자가 납득하도록 설명하는 것입니다.

- 한두 문장으로 씁니다.
- **참여자의 발화를 그대로 인용하지 마십시오.** 참여자는 방금 그 대화를 했으므로 자기 말을 다시 읽을 필요가 없습니다.
- 대신 참여자가 **무엇을 중요하게 여겼기에** 이 판정에 이르렀는지를 서술하십시오. 인터뷰에서 드러난 우선순위와 선택을 짚어 주십시오.
- 참여자를 2인칭으로 부르며 담백하게 씁니다. 성격을 진단하는 말투를 쓰지 마십시오. 이것은 참여자가 바라는 **도시**에 대한 판정입니다.
- 축 이름이나 반대 극의 글자를 설명에 늘어놓지 마십시오. 판정된 글자가 뜻하는 바를 자연스러운 말로 풀어 쓰십시오.

예를 들어 J로 판정된 축이라면, 참여자가 예측 가능한 질서를 자유로운 혼합보다 앞세웠다는 점과 그렇게 판단한 계기를 짚습니다.

### axis_demands

네 개 축 각각에 대해, 참여자가 도시에 요구하는 바를 정리합니다.

축은 `EI`, `SN`, `TF`, `JP` 네 개이며 이 순서로 배열합니다. 각 축마다 요구를 1~2개 만드십시오.

요구 하나는 다음으로 구성합니다.

- `id`: `축-D순번` 형식. 예를 들어 `EI-D1`, `EI-D2`, `SN-D1`
- `title`: 그 요구가 무엇인지 한눈에 보이는 짧은 제목. 명사구로 쓰십시오
- `description`: **문장 배열**입니다. 한 문장씩 따로 담으십시오. 참여자가 이 중 마음에 들지 않는 문장을 골라 고쳐 달라고 할 수 있기 때문에, 한 항목에 여러 문장을 뭉쳐 넣지 마십시오. 2~4문장이 적당합니다
- `quotes`: 이 요구의 근거가 된 참여자 발화. 각 항목은 `text`(원문 그대로)와 `turn`(턴 번호)을 가집니다. 1~2개

`description`은 참여자가 무엇을 원하고 왜 원하는지를 도시에 대한 요구의 언어로 적으십시오. "~하면 좋겠다는 바람을 밝혔습니다" 같은 전언체가 아니라, 요구 자체를 서술하십시오.

축의 이름이나 극의 글자를 `description`에 노출하지 마십시오. 참여자는 축이 무엇인지 모릅니다.

## 수정 요청을 받은 경우

참여자가 특정 문장을 고르고 의견을 남기면, 그 의견을 반영해 **`axis_demands` 네 축 전체를 처음부터 다시 만드십시오.**

- 참여자가 고른 문장과 그 의견이 입력으로 주어집니다.
- 참여자의 의견은 명령입니다. 그가 아니라고 한 것은 아닌 것입니다. 압축본보다 참여자의 수정 의견을 우선하십시오.
- 고쳐 달라고 한 부분만 손대고 나머지를 그대로 복사하는 것이 아니라, 수정된 이해를 바탕으로 네 축의 요구를 **전부 다시 씁니다.** 참여자의 한마디가 다른 축의 해석까지 바꿀 수 있기 때문입니다.
- 다만 위의 절대 규칙은 그대로입니다. 참여자가 의견에서 말하지 않은 것을 지어내지 말고, 인용은 여전히 원문 그대로여야 합니다.
- `summary`는 바꾸지 마십시오. 받은 그대로 되돌려 주십시오.

## 출력 형식

다음 스키마의 JSON **객체 하나만** 출력하십시오. 코드블록이나 설명 문장으로 감싸지 마십시오.

```
{
  "self_info": {
    "nickname": "string",
    "region": "string",
    "dream_or_job": "string"
  },
  "summary": ["string", "string"],
  "axis_reasons": [
    { "axis": "EI", "reason": "string" },
    { "axis": "SN", "reason": "string" },
    { "axis": "TF", "reason": "string" },
    { "axis": "JP", "reason": "string" }
  ],
  "axis_demands": [
    {
      "axis": "EI",
      "demands": [
        {
          "id": "EI-D1",
          "title": "string",
          "description": ["string", "string"],
          "quotes": [{ "text": "string", "turn": 3 }]
        }
      ]
    },
    { "axis": "SN", "demands": [] },
    { "axis": "TF", "demands": [] },
    { "axis": "JP", "demands": [] }
  ]
}
```

`axis_reasons`와 `axis_demands`에는 네 축이 모두 `EI`, `SN`, `TF`, `JP` 순서로 있어야 하며 각 축의 `demands`는 비어 있으면 안 됩니다. 축의 판정 글자는 코드가 따로 채우므로 여기에 넣지 마십시오.

수정 요청을 받은 경우에는 `axis_reasons`도 바꾸지 마십시오. 판정은 요구 수정으로 달라지지 않습니다.
