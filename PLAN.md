> 문서 성격: 아무 맥락이 없는 코딩 LLM이 이 문서 하나만 읽고 구현할 수 있도록 만든 프로젝트 최상위 설계 문서이자 단일 진실 원천(SSOT).
>
> 상태: **Phase 3까지 완료. 핵심 인터뷰 MVP가 실제로 동작한다.** 참여자가 인터뷰를 완주하고, 코드가 유형을 판정하고, LLM이 개인 보고서를 만들고, 참여자가 요구를 수정하고 제출한 뒤 유형 카드를 이미지로 공유하는 흐름 전체가 실제 Gemini로 돌아간다. 다음 작업은 Phase 4이며 시작하려면 GCP 프로젝트가 필요하다. 자세한 진행 상황은 12장 끝의 "현재 진행 상황"을 본다.

> 최종 기준일: 2026-07-15.

## 1. 프로젝트 성격과 목표

유스플랜AI는 인천시 청년정책 아이디어 공모전에서 심사위원에게 시연할 프로토타입이다. 대규모 상용 서비스나 실제 정부 운영 시스템이 아니다.

모든 구현 판단에는 다음 우선순위를 적용한다.

- 핵심 기능의 큰 틀을 완성한다.
- 깔끔한 UX/UI를 만든다.
- 시연 중 안정적으로 동작하게 한다.
- 완벽함보다 끝까지 작동하는 전체 사용자 여정을 우선한다.
- 과도한 보안·암호화, 대규모 트래픽 대응, 완벽한 예외 처리는 구현하지 않는다.

도시기본계획은 장기 도시 발전 방향을 정하지만, 기존 공청회는 시간·정보·접근성 문제로 청년의 참여를 어렵게 한다. 유스플랜AI는 공청회를 대체하지 않고 보완한다.

참여자는 앱 설치 없이 모바일 웹에 접속해 LLM과 1:1로 대화한다. 2040년 인천에서의 자기 하루를 상상하고, AI의 꼬리질문을 통해 막연한 생각을 구체적인 정책 수요와 문제의식으로 발전시킨다. 시스템은 대화를 구조화된 보고서와 "내가 바라는 2040년 인천의 도시 유형"으로 변환한다.

도시 유형은 MBTI의 4글자 형식을 빌린 16유형이다. 참여자에게는 참여 보상과 SNS 공유 소재가 되고, 담당자에게는 축별 분포와 근거 발화를 함께 제공한다. 자체 AI 모델은 개발하지 않고 상용 Gemini API를 사용한다.

성공 조건은 다음과 같다.

- 시연에서 `접속 → 인터뷰 → 유형·보고서 확인/수정 → 제출 → 공유 카드 → 담당자 화면`이 끊김 없이 작동한다.
- 참여자가 모바일 웹에서 인터뷰를 완주하고 결과가 자기 생각을 반영한다고 느낀다.
- 담당자가 축별 분포, 평균 강도, 정성 요약, 근거 인용을 함께 볼 수 있다.
- 7~8월 약 10명의 시범운영에서 질문 구성, 응답 품질, 축 커버리지, 유형 납득도, 요약 정확도, 비식별화를 실측한다.

## 2. 확정 결정 요약

### 2.1 제품 및 설계 원칙

- 핵심 MVP는 `AI 인터뷰 → 유형 판정·보고서 구조화 → 참여자 확인·제출`이다.
- 시스템과 백엔드·프론트엔드의 틀을 먼저 완성하고, 프롬프트·축·유형 문구는 외부 파일 튜닝으로 정련한다.
- 인터뷰·유형 판정·확인·제출은 실시간 구간이다.
- 다수 제출본의 비식별화·집계·정성 요약은 제출 후 실행하는 배치 구간이다.
- 실시간 구간과 배치 구간의 경계는 제출 액션이다.
- 유형 정량화에는 사전 정의된 4축을 사용한다.
- 임베딩 군집화는 통제와 해석이 어려우므로 사용하지 않는다.
- 유형은 심리검사가 아니며 성격 MBTI와 완전히 분리한다.
- 성격 MBTI를 수집하거나, 도시 유형과 병렬 제시하거나, 상관관계를 검증하지 않는다.
- 프롬프트, 루브릭, 축 정의, 채점 앵커는 코드에 하드코딩하지 않는다.
- 기능 변경은 먼저 이 문서에 반영한다.

### 2.4 UI 디자인 원칙

참여자 화면은 **모바일 전용**으로 설계한다. QR로 접속한 휴대폰이 유일한 실사용 환경이기 때문이다. 데스크톱 브라우저로 열면 모바일 폭을 유지한 채 화면 가운데에 배치한다. 반응형으로 넓은 화면용 레이아웃을 따로 만들지 않는다. `/admin`은 예외이며 데스크톱에서 본다.

디자인 방향은 토스(Toss)의 결을 따른다. 구체적으로 다음을 뜻한다.

- 한 화면에는 하나의 목표만 둔다. 참여자가 다음에 무엇을 해야 하는지 고민하지 않아야 한다.
- 여백을 넉넉히 쓴다. 요소를 빽빽하게 채우지 않는다.
- 제목은 크고 굵게, 본문은 담백하게 쓴다. 타이포그래피가 위계를 만든다.
- 주요 행동 버튼은 화면 폭을 꽉 채우고 충분히 높게 만들어 엄지로 누르기 쉽게 한다.
- 모서리는 부드럽게 굴리고 그림자는 절제한다. 경계선보다 배경 대비로 영역을 구분한다.
- 문구는 짧고 사람의 말처럼 쓴다. 행정 용어를 쓰지 않는다. 모호한 비유로 에두르지 말고, 지금 무엇을 보고 있는지 그대로 말한다.
- 화면이 사용자의 조작을 이기지 않는다. 인터뷰 대화 영역은 참여자가 아래쪽에 있을 때만 새 글자를 따라 내려가고, 참여자가 위로 스크롤해 지난 대화를 읽는 동안에는 화면을 끌어내리지 않는다. 답변은 그동안에도 아래에서 계속 생성된다.

색은 **인천광역시 전용색상만** 사용한다. 임의의 브랜드 색을 만들지 않는다.

- Incheon Blue, Pantone 300C, `#005EB8`: 주조색. 주요 버튼, 강조, 참여자 말풍선.
- Incheon Green, Pantone 326C, `#00B2A9`: 보조색. 완료와 긍정 상태, 유형 결과의 강조.
- Incheon Gray, Pantone 425C, `#54585A`: 보조 텍스트와 비활성 상태.

본문 텍스트의 진한 먹색과 배경의 옅은 회색처럼 위 3색으로 표현할 수 없는 중립색은 무채색 계열에서 파생해 쓰되, 유채색 포인트는 위 3색을 벗어나지 않는다.

예외는 하나뿐이다. **본인인증 버튼에만 `#EB4D51`을 쓴다.** 본인인증은 인천시의 서비스가 아니라 통신사 인증 절차로 넘어가는 행동이고, 국내 이용자에게 이 계열의 붉은색이 본인인증의 관례적 신호로 통하기 때문이다. 이 색은 그 버튼 밖으로 번지지 않는다.

한글 서체는 Pretendard Variable을 애플리케이션에 번들로 포함해 사용한다. 외부 폰트 CDN에 런타임 의존하지 않는다.

## 3. 방법론과 전체 아키텍처

Anthropic의 AI Interviewer 방법론을 다음 세 단계로 적용한다.

1. Planning: 연구 질문과 탐색 주제로 인터뷰 루브릭을 만든다.
2. Interviewing: 참여자별 실시간 적응형 인터뷰를 진행한다.
3. Analysis: 원 방법론의 창발적 군집화 대신 사전 정의된 도시가치 축으로 정량화한다.

시스템은 참여자 여정에 따라 다섯 파트로 나눈다. 인터뷰의 공간 범위는 인천광역시 전체이고, 주제 범위는 인천시민 전체의 삶이다. 시범운영 참여자는 청년이다.

### 3.1 파트 A: 접속과 자격 확인

진입 순서는 다음과 같다.

1. QR 또는 링크 접속
2. 시스템 개요 확인
3. 개인정보 수집·이용 동의
4. 본인인증 목업
5. 인터뷰 시작

개인정보 동의는 별도 화면이 아니라 시작 화면 안의 고지 문구와 체크박스로 통합했다. 참여자가 체크해야 본인인증 버튼이 활성화된다.

실서비스에서는 휴대폰 본인인증으로 1인 1회 참여를 확인하고 출생연도를 함께 받는다. 프로토타입에서는 실제 인증 없이 버튼으로 통과시키며, 출생연도 입력 필드 하나로 인증 응답값을 대신한다.

프로토타입의 목업 인증은 중복참여를 실제로 강제하지 않는다. 이는 시연용 구현의 한계다. 출생연도는 1900~2026 범위의 정수인지 여부만 검사한다.

휴대폰 신원은 중복참여 방지 게이트로만 사용한다. 인터뷰 데이터에는 출생연도만 전달하고 다른 신원 정보는 결합하지 않는다.

참여는 열린 참여이고 무보상이다. QR 홍보물에는 사업 배경과 기대효과를 적는다. 입력은 타이핑, 언어는 한국어로 한정한다.

자동저장은 없다. 이어지는 범위는 브라우저가 페이지를 백그라운드에서 유지하는 정도에 따른다.

### 3.2 파트 B: AI 인터뷰 엔진

참여자가 2040년 인천에서의 하루를 상상하도록 돕고, 꼬리질문으로 정책 수요를 구체화한다. 동시에 4개 도시가치 축을 판정할 수 있을 만큼 모든 축을 탐색한다.

출생연도는 세션 생성 시 백엔드로 전달한다. 백엔드는 연 단위 추정 나이(`2040 − 출생연도`)를 계산해 시스템 프롬프트에 주입한다.

별명, 거주 지역, 꿈 또는 현재 직업은 별도 폼으로 받지 않는다. LLM이 인터뷰 도입부에서 라포 형성을 겸해 자연스럽게 질문하며, 종료 후 구조화 단계에서 대화록으로부터 추출한다.

단일 시스템 프롬프트가 도입, 심화, 정리 전 단계를 관장한다. 백엔드가 턴 안내나 미커버 축 힌트를 주입할 때에는 사용자 발화 뒤에 `[운영 지시: ...]` 블록을 덧붙인다. `system.md`에는 이 블록이 참여자 발화가 아니라 백엔드의 운영 지시이며, 참여자의 의견이나 증거로 해석하면 안 된다고 명시한다.

심화 루브릭은 4축에서 파생된 주제 블록으로 구성한다. 축당 2~3턴 안팎을 가이드로 삼지만 강제하지 않는다. 시스템 프롬프트는 다음을 지켜야 한다.

- 자연스러운 대화를 유지하면서 4개 축을 모두 탐색한다.
- 4개 축 탐색 전에는 정상 종료 토큰을 출력하지 않는다.
- 마무리 전 미탐색 축이 있으면 해당 주제를 질문한다.

백엔드는 누적 증거로 축별 커버 여부를 판단한다. 커버 기준은 **증거 2건 이상**이다. 증거가 0건인 축은 아직 다루지 않은 축이고, 1건인 축은 스치고 지나간 축이다. 둘 다 미커버로 보고 "아직 S/N 관련 대화가 없음" 또는 "S/N 축의 증거가 부족하니 더 탐색할 것"과 같은 내용을 `[운영 지시: ...]` 블록에 넣어 다음 사용자 발화 뒤에 붙인다. 별도 판정 호출은 하지 않는다.

커버 기준을 1건이 아니라 2건으로 두는 이유는 강도 계산 때문이다. 어떤 축의 증거가 한 건뿐이면 그 극의 점수가 축 총점과 같아져 강도가 100%로 계산된다. 참여자가 한 번 스친 이야기가 100% 확신으로 표시되는 것은 납득하기 어렵다. 축마다 증거를 여러 건 쌓아야 강도가 의미를 갖는다. 2는 백엔드 설정값으로 두고 시범운영에서 튜닝한다.

한 턴은 `사용자 발화 1회 + AI 응답 1회`다.

- 목표는 15턴이다.
- 12턴 전에는 마무리하지 말라는 운영 지시를 매 턴 추가한다. 4축을 형식적으로 한 번씩만 찍고 조기 종료하면 증거가 축당 한두 건에 그쳐 유형 판정이 빈약해지기 때문이다.
- 12턴째부터는 반대로 남은 턴 안에 자연스럽게 마무리를 준비하라는 운영 지시를 매 턴 추가한다.
- 강제 상한은 없다. 15턴 이후에도 안내만 계속한다.
- 15와 12는 백엔드 설정값으로 두고 시범운영에서 튜닝할 수 있게 한다.

조기 종료 방지는 안내일 뿐 강제가 아니다. 참여자가 그만하겠다고 말하면 12턴 전이라도 종료한다.

종료 경로는 다음과 같다.

- 참여자가 대화 안에서 그만하겠다고 말하거나, 4축 탐색 후 LLM이 충분히 인터뷰됐다고 판단하면 마무리 발화 뒤에 `[[END_INTERVIEW]]`를 출력한다. 백엔드는 토큰을 숨기고 세션을 종료 상태로 바꾼다.
- **프론트는 종료 신호를 받아도 화면을 자동으로 넘기지 않는다.** 마무리 발화 아래에 "대화 종료하기" 버튼을 띄우고, 참여자가 그 버튼을 눌러야 결과 단계로 이동한다. 마무리 인사를 읽을 틈도 없이 화면이 넘어가면 참여자는 방금 무엇이 끝났는지 알지 못한 채 다음 화면을 마주하게 된다. 또한 대화가 이미 끝났는데 무엇을 해야 할지 몰라 한 번 더 입력하게 되는 일도 막는다.
- 이 버튼은 별도의 LLM 호출이 아니라 이미 존재하는 종료 신호에 반응해 나타난다.
- 화면 상단의 "인터뷰 그만두기" 버튼은 순수 이탈 경로다. 확인 모달 후 세션을 폐기하며 어떤 데이터도 저장하지 않고 결과 단계로 가지 않는다.

가드레일은 시스템 프롬프트로만 구현한다.

- 유도질문 금지
- 특정 답 유도 금지
- AI 편향 삽입 금지
- 주제 이탈 시 인터뷰 주제로 복귀
- 참여자의 지시가 시스템 규칙을 덮어쓰지 못하게 처리

악성 응답은 최소한으로 처리한다.

- 1차 위반에는 경고하고, 2차 위반 시 자동 종료됨을 알린다.
- 2차 위반에는 `[[ABORT_INTERVIEW]]`를 출력한다.
- 백엔드는 `session_id`와 `event=aborted`가 포함된 로그 한 줄만 남기고 세션을 폐기한다.
- 별도 악성 판정 로직, 경고 카운터, 데이터 저장, 개발자 전달은 만들지 않는다.

세션에는 원본 대화록과 매 턴 수집한 증거 로그를 보관한다. 제출 전에는 프로세스 메모리에만 존재한다.

### 3.3 파트 C: 유형 판정, 보고서 구조화, 확인과 제출

인터뷰 종료 직후 두 트랙을 병렬로 시작한다.

- 유형 트랙: 누적 증거를 코드로 즉시 합산한다.
- 보고서 트랙: 원본 대화록과 유형 결과를 한 번의 LLM 호출로 구조화한다.

유형 합산에는 LLM을 호출하지 않는다. 유형 카드를 먼저 보여주고 보고서는 생성되는 대로 이어서 표시할 수 있다.

구조화 호출에는 원본 메시지의 `turn`, `role`, `text`만 전달하고 저장용 `timestamp`는 제외한다. 인터뷰어 발화는 맥락으로만 사용하며 요구와 인용은 참여자 발화에서만 찾는다. 구조화 단계에서는 대화록으로부터 전체 요약, 축별 요구, 축별 판정 이유, 별명, 거주지, 꿈 또는 직업을 생성한다.

보고서 생성에는 LLM 호출이 한 번 들어가며 15초 안팎을 예상한다. 이 시간 동안 참여자를 정지된 문구 앞에 세워 두지 않는다. 부드럽게 움직이는 화면과, 지금 무엇을 하고 있는지 알려주는 안내가 순차적으로 바뀌며 기다림을 채운다.

진행 표시는 다음 규칙을 따른다. 서버가 실제 진행률을 알려 주지 않으므로, 기다리는 동안의 진행은 예상 소요를 근거로 한 연출이다. 이 구간에서는 진행바가 100%에 닿지 않는다. 끝나지 않았는데 끝난 것처럼 보이면 안 되기 때문이다.

**응답이 실제로 도착하면 진행바를 100%까지 채우고 잠깐 머문 뒤 결과 화면으로 넘어간다.** 이때의 100%는 거짓이 아니라 사실이다. 다 채우지 않은 채로 화면이 넘어가면 참여자는 무언가 덜 끝난 채로 넘어갔다고 느낀다. 완료를 눈으로 확인시켜 준 뒤에 넘긴다.

결과는 한 화면에 몰아넣지 않고 세 단계로 나눈다. 한 화면에 모두 담으면 스크롤이 지나치게 길어져 참여자가 무엇을 해야 하는지 알기 어렵다. 각 화면은 하나의 목적만 갖는다.

**결과 1단계 — 내 유형 확인**

- 4글자 유형 코드, 유형 별명, 한 줄 설명, 축별 강도 바를 담은 유형 카드
- 공유하기 버튼과 이미지로 저장하기 버튼
- 축별 판정 이유
- 다음 단계로 넘어가는 버튼

**결과 2단계 — 내 이야기와 요구 확인**

- 인터뷰 전체 요약
- 축별 요구 보고서
- 제출 버튼
- 수정하거나 의견을 덧붙이고 싶은지 묻는 안내와 그 진입점

**결과 3단계 — 요구 수정**

- 요구 문장 선택과 참여자 의견 입력
- 4축 전체 재생성

판정 근거로는 참여자의 발화를 다시 인용하지 않는다. 대신 각 축에 대해 **왜 그 글자로 판정했는지를 한두 문장으로 설명한다.** 방금 나눈 대화의 내용을 그대로 되돌려주는 것은 참여자에게 새로운 정보가 아니다. 참여자가 알고 싶은 것은 자기가 한 말이 아니라 그 말이 어떻게 이 판정으로 이어졌는가다.

판정 이유는 증거와 판정 결과를 근거로 LLM이 생성한다. 유형 코드와 강도가 코드로 결정된 뒤에 만들어지므로 판정과 어긋나지 않는다. 유형 결과와 마찬가지로 참여자가 수정할 수 없으며 요구를 수정해도 바뀌지 않는다.

수정 가능한 영역은 축별 요구뿐이다. 유형 코드, 축별 강도, 판정 근거, 전체 요약은 고정한다. 수정 LLM은 `axis_demands`만 출력하고, 서버가 기존 `summary`, `axis_reasons`, `self_info`와 메타데이터를 복사해 전체 개인 보고서를 조립한다.

축별 요구 설명은 문장 배열로 표시한다. 참여자가 하나 이상의 문장을 선택하고 의견을 입력하면, 선택 문장과 참여자 의견을 재생성 지시의 입력으로 전달한다. LLM은 일부 요구만 수정하지 않고 `axis_demands`의 4축 전체를 기존과 동일한 스키마로 재생성한다. `summary`와 `type_result`는 변경하지 않는다. 사용자가 텍스트를 직접 편집하지 않는다.

수정 횟수에는 제한이 없다. 15회에 도달하면 제출을 권하는 문구만 표시하고 수정은 차단하지 않는다.

제출 시 원본 대화록, 증거 로그, 유형 결과, 최종 보고서, 메타데이터를 Firestore에 영속화하고 메모리 세션을 제거한다. 한 제출 흐름은 1회 제출로 끝나며 재작성과 재제출은 허용하지 않는다. 목업 인증은 다른 세션을 통한 중복참여까지 실제로 차단하지 않는다.

공유 카드는 프론트가 결과 단계에서 받아둔 `type_result` 상태로 렌더링한다. 서버에서 다시 조회하지 않으며 `submit` 응답은 `submission_id`만 반환한다.

공유와 이미지 저장은 **제출 전 결과 1단계에서도 제공한다.** 유형 카드는 참여자가 가장 먼저 보고 싶어 하는 것이고, 그것을 보기 위해 제출을 먼저 요구하면 마찰이 생긴다. 제출하지 않고 카드만 공유한 뒤 이탈할 수 있다는 뜻이지만, 제출을 인질로 삼는 것보다 낫다고 본다. 제출 완료 화면에도 같은 공유 수단을 유지한다.

모바일에서는 Web Share API 공유 시트를 연동한다. 새로고침하거나 재접속하면 결과를 다시 볼 수 없다는 안내도 표시한다.

### 3.4 파트 D: 후처리 분석 파이프라인

담당자 웹의 "분석 실행" 버튼이 파이프라인을 한 번 실행한다. 파트 D는 참여자 흐름과 분리된 배치 작업이며, 실행 자체는 약 10건 규모의 admin HTTP 요청 안에서 동기 완료한다.

제출 응답이 끝난 뒤 백그라운드에서 Gemini가 요구 텍스트와 인용 발화의 이름·연락처를 블라인드 처리해 제출본의 비식별 사본을 채운다. 실패하면 로그만 남기고 제출 자체는 유지한다.

분석 실행 단계는 다음과 같다.

1. 비식별 사본이 없는 제출본만 Gemini로 보충하고 기존 비식별본은 재사용한다.
2. 코드가 축별 극 인원수, 극별 평균 강도, 16유형 분포를 계산한다.
3. Gemini가 각 축과 극의 요구 경향을 요약하고 대표 인용을 선정해 인용 ID를 부여한다.

규칙 기반 비식별 필터, 최소 인원 임계값, 임베딩, 군집화는 사용하지 않는다. 특정 회사·지역 같은 준식별자의 처리 기준은 베타 테스트에서 정한다.

분석을 다시 실행해도 이미 있는 비식별 사본은 다시 만들지 않는다.

### 3.5 파트 E: 담당자 웹과 보고서

같은 Next.js 앱의 `/admin` 경로에 개발자·담당자 화면을 만든다. 실제 작동 화면이어야 하지만 로그인이나 권한 시스템은 구현하지 않는다. 링크를 외부에 공개하지 않는 것으로 접근을 통제한다.

종합 리포트에는 정확히 다음 다섯 요소가 있어야 한다.

1. 축별 분포와 극별 평균 강도
2. 16유형 분포
3. 축별·극별 정성 요구 요약
4. 인용 ID가 붙은 비식별 대표 인용
5. 축과 유형의 의미를 설명하는 담당자용 정적 해설

담당자용 축·유형 해설은 `web/`의 정적 자산으로 둔다. 구체적인 파일 위치는 Phase 6 착수 시 결정한다.

추가 화면은 다음과 같다.

- 제출본 목록
- 제출본 상세
- 유형 결과
- 개인 보고서
- 원본 대화록
- 분석 실행 버튼

종합 리포트와 대표 인용은 비식별본을 사용한다. 인용 ID로 제출본까지 근거를 추적할 수 있게 한다. 담당자가 간접 정보를 조합해 개인을 추적하지 않는다고 전제하며, 그 이상의 익명성 검증은 구현 범위에 포함하지 않는다.

## 4. 데이터 흐름과 런타임 계약

데이터는 다음 순서로 단방향 이동한다.

1. 메모리 세션에 원본 대화록과 증거 로그 누적
2. 인터뷰 종료
3. 코드로 유형 결과 산출
4. 원본 대화록과 유형 결과로 축별 요구 보고서 생성
5. 참여자가 요구 영역 확인·수정
6. 제출
7. Firestore 영속화
8. 응답 후 백그라운드로 비식별 사본 생성
9. 분석 실행 시 누락된 비식별 사본 보충
10. 코드 집계와 LLM 정성 요약
11. 담당자 리포트 제공

제출 전 데이터는 FastAPI 프로세스 메모리의 임시 상태다. 서버 재시작, 참여자 이탈, 악성 종료로 사라져도 허용한다. 이탈·악성 종료를 포함해 제출되지 않은 인터뷰는 저장하지 않는다. 악성 종료만 이벤트 로그 한 줄을 남긴다.

### 4.1 프론트엔드와 백엔드의 책임

Next.js는 사용자가 보는 모든 화면을 담당한다.

- 진입과 동의
- 본인인증 목업
- 인터뷰 UI
- 스트리밍 표시와 타자기 효과
- 유형 카드
- 보고서 확인·수정
- 공유 카드
- `/admin`

FastAPI는 다음을 담당한다.

- 인터뷰 세션
- Gemini 호출
- 프롬프트 조립
- 증거 수집
- 유형 합산
- 대화록 구조화
- Firestore 저장
- 비식별화와 집계
- 담당자용 데이터 제공

프론트는 백엔드 HTTP API만 호출한다. Gemini 키, 프롬프트, 축 정의, 도메인 지식은 백엔드에만 둔다. 유형 별명·설명만 표시용 프론트 정적 자산으로 둔다.

### 4.2 SSE 스트리밍과 머신 트레일러

FastAPI는 Gemini 스트리밍 응답을 SSE로 중계하고 Next.js는 fetch 스트림으로 소비한다.

SSE 이벤트는 두 종류다.

- `delta`: 참여자에게 표시할 응답 텍스트 조각. 데이터는 `{"text": ...}`다.
- `end`: 스트림 종료와 세션 상태 플래그. 데이터는 `{"state": ...}`이고 값은 `continue`, `ended`, `aborted` 중 하나다.

인터뷰 LLM의 모든 어시스턴트 응답은 첫 인사를 포함해 다음 순서를 반드시 따른다.

1. 참여자에게 보이는 텍스트
2. 줄바꿈
3. `[[EVIDENCE]]` 바로 뒤에 래퍼 객체가 없는 JSON 배열을 붙인 한 줄
4. 종료 턴이면 다음 줄에 `[[END_INTERVIEW]]` 또는 `[[ABORT_INTERVIEW]]`를 단독으로 둔 한 줄

모든 어시스턴트 응답에 EVIDENCE 줄이 필수다. 증거가 없으면 `[[EVIDENCE]][]`를 출력한다. 첫 인사는 직전 사용자 발화가 없으므로 반드시 `[[EVIDENCE]][]`를 출력한다. 정상 진행 턴에는 EVIDENCE 줄에서 응답이 끝나며 종료 제어 토큰을 붙이지 않는다.

EVIDENCE JSON은 반드시 배열 자체여야 하며 래퍼 객체를 사용하지 않는다. JSON 배열 전체와 `[[EVIDENCE]]` 표지는 같은 한 줄에 있어야 한다. 종료 제어 토큰은 EVIDENCE 줄 다음 줄에 다른 텍스트 없이 단독으로 출력한다.

백엔드는 스트림에서 `[[`를 감지하면 이후 출력을 버퍼링하고 트레일러 전체를 프론트에 보내지 않는다. 증거는 세션에 누적하고 제어 토큰은 세션 상태 전환에 사용한다.

트레일러 파싱에 실패하면 해당 턴의 증거 전체를 버리고 실패 로그만 남긴다. 불량 증거를 합산에 포함하지 않는다.

## 5. 도시가치 유형화 계약

### 5.1 성격과 표기

진단 대상은 참여자의 성격이 아니라 참여자가 바라는 2040년 인천의 수요와 지향이다. 심리측정 검사가 아니라 참여 흥미와 정책 전달력을 높이기 위한 진단 장치다.

MBTI와 같은 E/I, S/N, T/F, J/P 글자를 사용하지만 도시가치 문맥으로 재정의한다. 4개 축을 조합해 16개 유형을 만든다.

축 설계는 인천·서울·부산의 2040 도시기본계획에서 반복되는 도시가치 대립과 AURI 지역 Space-MBTI 등의 선례를 참고한다. "현재 유형"이 아니라 "바라는 유형"을 표현한다.

### 5.2 확정된 4축 v1

#### E/I: 에너지의 방향

- E, 개방·교류·활력: 관문, 글로벌, 번화, 외부 개방
- I, 정주·정온·내밀: 조용한 동네, 프라이버시, 일상 밀착

성격 MBTI의 외향적 자극과 내향적 정온이라는 개념적 결만 대응시킨다.

#### S/N: 시선의 대상

- S, 현실·실용·기존 자원: 원도심, 역사자원, 검증된 인프라 활용
- N, 미래·혁신·첨단: 신도시, 스마트, 실험

개항장 근대건물 재활용은 S 근거가 될 수 있고, 이를 새로운 작업공간으로 재해석하려는 발화는 N 요소가 될 수 있다.

#### T/F: 우선하는 가치

- T, 효율·기능·경쟁력: 생산, 최적화, 산업
- F, 관계·돌봄·정서: 공동체, 복지, 포용, 소속

성격 MBTI의 논리·효율과 관계·정서라는 개념적 결만 대응시킨다.

#### J/P: 도시의 운영 방식

- J, 계획·정돈: 성장관리, 예측 가능성, 정연한 용도
- P, 유연·혼합·자생: 용도 혼합, 자생적 활력

성격 MBTI의 계획·질서와 유연·개방이라는 개념적 결만 대응시킨다.

개발과 보전의 대립은 별도 축으로 만들지 않는다. 주로 S/N과 J/P에 흡수한다.

축 정의와 축·극·강도별 예시 발화인 채점 앵커는 `api/prompts/axes.md`를 단일 원천으로 삼는다. 인터뷰 루브릭과 증거 태깅이 이 파일을 함께 참조한다. 같은 파일의 기계 판독 JSON 블록(축 순서·극·기본 극·표시명)은 백엔드 코드 상수(scoring·trailer·prompts)의 단일 원천이기도 하며 `api/app/axes.py`가 이를 파싱한다.

### 5.3 증거 기반 판정

대화록 전체를 한 번에 평가하는 홀리스틱 LLM 판정은 실행마다 기준이 흔들릴 수 있으므로 사용하지 않는다. 개방형 대화에서 미시 증거를 추출하고 코드는 이를 결정적으로 합산한다.

인터뷰 질문을 검사형 선택 질문으로 바꾸지 않는다. "E와 I 중 어느 쪽인가"처럼 범주를 직접 선택시키지 않는다.

축의 두 극을 선택지로 나란히 제시하는 질문도 같은 금지에 포함한다. 축 이름을 말하지 않더라도 "활기찬 동네가 좋으세요, 조용한 동네가 좋으세요"처럼 물으면 사실상 검사형 선택 질문이다. 참여자가 자기 언어가 아니라 축의 언어로 답하게 되고, 그렇게 얻은 답은 참여자의 생각이 아니라 인터뷰어가 제시한 프레임에 대한 반응이다. 열린 질문으로 장면을 묻고, 증거는 참여자가 스스로 꺼낸 말에서만 찾는다.

LLM은 매 턴 직전 사용자 발화에서 축 관련 증거를 찾는다. 증거 한 건은 다음 필드로 구성한다.

- 축
- 극
- 가중치
- 사용자 발화 원문 인용
- 턴 번호

가중치는 다음과 같다.

- 1: 약한 증거
- 2: 보통 증거
- 3: 강한 증거

증거가 없는 턴은 빈 배열을 출력한다. 채점 기준은 `axes.md`의 채점 앵커로 고정한다.

인터뷰 종료 시 코드는 각 축의 두 극에 대해 증거 가중치 합을 계산한다.

- 극 점수는 해당 극 증거의 가중치 총합이다.
- 점수가 큰 극을 판정 글자로 정한다.
- 원시 강도 비율 `x`는 `승자 점수 ÷ 축 총점 × 100`으로 계산한다.
- 정수 강도는 파이썬 기본 `round`를 사용하지 않고 `floor(x + 0.5)` 방식으로 반올림한다.
- 반올림한 값이 51 미만이면 51로 클램프한다.
- 두 극의 표시 백분율 합은 100이고 동률은 표시하지 않는다.

점수가 동률이면 다음 순서로 결정한다.

1. 증거 개수가 더 많은 극
2. 증거 개수도 같으면 기본 극

기본 극은 다음과 같다.

- E/I는 I
- S/N은 S
- T/F는 F
- J/P는 J

해당 축의 증거가 하나도 없으면 기본 극을 51%로 판정하고 로그를 남긴다. 빈 축은 원칙적으로 커버리지 규칙이 막아야 하며, 발생률은 시범운영 지표로 측정한다.

각 축의 판정 출력은 다음을 포함한다.

- 판정 글자
- 강도 51~100
- 두 극의 가중치 점수
- 발화 원문, 턴, 가중치가 포함된 증거 목록

유형은 네 축의 판정 글자를 이어 붙여 만든다. 참여자는 유형, 강도, 증거를 수정할 수 없다.

증거 태깅 자체는 LLM 판단이므로 완전히 재현되지는 않는다. 다만 한 번의 큰 판단 대신 여러 미시 판단을 합산하고, 합산·반올림·동률·빈 축 처리를 코드로 고정해 변동 범위를 줄인다.

### 5.4 산출물

참여자 산출물은 다음과 같다.

- 4글자 도시 유형
- 유형 별명과 한 줄 설명
- 축별 강도
- 축별 판정 이유
- 축별 요구 보고서
- 이미지로 저장하고 공유할 수 있는 유형 카드

참여자 화면에 보이는 축 극 라벨은 5.2의 축 정의와 별개로 일상어 표시 문구를 쓰며 `web/src/lib/city-axes.ts`가 그 원천이다. 5.2의 축 정의, `axes.md`의 채점 앵커와 증거 태깅 기준은 이 표시 문구의 영향을 받지 않는다.

담당자 산출물은 다음과 같다.

- 축별 인원 분포
- 극별 평균 강도
- 16유형 분포
- 축별·극별 정성 요약
- 비식별 대표 인용
- 축과 유형의 의미 설명

16유형 콘텐츠는 `web/src/data/city_types.json`에 둔다. 각 항목의 스키마는 다음 세 필드로 고정한다.

- `code`
- `nickname`
- `description`

초안은 Claude가 작성하고 팀이 검수·확정한다. 유형 카드도 이 파일의 별명과 설명을 사용한다.

## 6. API 계약

운영 엔드포인트와 별개로 다음 두 개를 둔다.

- `GET /health`
  - 서비스 상태 확인
  - Cloud Run 배포 후에도 유지한다

- `POST /api/dev/echo`
  - Phase 0 검증 전용이다
  - 입력 텍스트를 Gemini에 한 번 보내고 응답과 사용 모델명을 돌려준다
  - `프론트 → 백엔드 → Gemini` 왕복이 실제로 연결되었음을 확인하는 것이 유일한 목적이다
  - Phase 1에서 `/api/sessions/*`가 생기면 제거한다

### 6.1 참여자 API

- `POST /api/sessions`
  - 입력: 본인인증 목업에서 받은 출생연도
  - 검증: 1900~2026 범위의 정수인지 검사
  - 출력: `session_id`

- `POST /api/sessions/{id}/start`
  - 기능: LLM의 첫 인사 생성
  - 트레일러: 직전 사용자 발화가 없으므로 `[[EVIDENCE]][]`
  - 출력: SSE 스트림

- `POST /api/sessions/{id}/messages`
  - 입력: 사용자 발화 텍스트
  - 출력: SSE 스트림

- `DELETE /api/sessions/{id}`
  - 기능: "인터뷰 그만두기" 이탈 처리
  - 결과: 세션 폐기, 저장 없음

- `POST /api/sessions/{id}/result`
  - 기능: 유형 코드 합산과 원본 대화록 직접 구조화 트랙을 병렬 실행
  - 출력: 유형 결과와 개인 보고서 JSON
  - 참여자 응답에서는 유형 결과의 `evidence`·`scores`와 보고서 요구의 `quotes`를 제외

- `POST /api/sessions/{id}/result/revise`
  - 입력: 하나 이상의 선택 문장 위치와 참여자 의견
  - 문장 위치: `axis`, `demand_id`, 문장 인덱스
  - 기능: 선택 문장과 의견을 입력으로 사용해 `axis_demands` 4축 전체를 동일한 스키마로 재생성
  - 출력: 4축 전체의 요구 영역이 재생성된 개인 보고서 JSON
  - `summary`와 `type_result`는 불변

- `POST /api/sessions/{id}/submit`
  - 기능: Firestore 영속화와 세션 종료
  - 출력: `submission_id`만 반환
  - 공유 카드: 프론트가 결과 단계에서 보관 중인 `type_result` 상태로 렌더링

### 6.2 담당자 API

- `GET /api/admin/submissions`
  - 제출본 목록

- `GET /api/admin/submissions/{id}`
  - 유형 결과, 개인 보고서, 원본 대화록을 포함한 제출본 상세

- `POST /api/admin/analysis/run`
  - 참여자 흐름과 분리된 분석 배치 파이프라인을 admin 요청 안에서 동기 실행

- `GET /api/admin/analysis/latest`
  - 최신 종합 리포트 조회

### 6.3 HTTP 오류 계약

- 존재하지 않는 세션에 대한 호출은 404를 반환한다.
- 현재 세션 상태에 맞지 않는 호출은 409를 반환한다.
- 그 외 서버 오류는 500을 반환한다.
- 자동 재시도와 복구 로직은 구현하지 않는다.

### 6.4 개발자 모드

결과 화면, 보고서, 유형 카드, 담당자 리포트를 확인하려면 그때마다 인터뷰를 완주해야 한다. 한 번에 십수 턴이 걸리므로 UI를 반복해 다듬는 동안 이 비용을 감당할 수 없다. 그래서 인터뷰 중간을 건너뛰는 개발자 모드를 둔다. 이 기능의 이름은 어디에서나 "개발자 모드"로 통일한다.

- `GET /api/dev/fixtures`
  - 사용 가능한 대화 묶음집 목록과 각 묶음집의 설명

- `POST /api/dev/sessions/{id}/load`
  - 입력: 대화 묶음집 이름
  - 기능: 진행 중인 세션의 대화록과 증거 로그를 저장된 묶음집의 것으로 교체하고, **세션을 곧바로 종료 상태로 바꾼다**
  - 출력: 주입된 원본 대화 메시지 배열. 화면이 대화 맥락을 이어서 보여주려면 백엔드가 돌려줘야 한다

개발자 모드는 세션을 새로 만들지 않고 이미 만들어진 세션에 대화를 주입한다. 참여자와 똑같이 시작 화면을 거쳐 인터뷰에 들어온 뒤, 대화 도중 원하는 시점에 완성된 대화를 불러오는 방식이다. 그래야 시작 화면과 세션 생성 경로가 개발자 모드 때문에 갈라지지 않는다.

대화 묶음집은 이미 완주된 인터뷰다. 불러온 뒤에 마무리 발화를 다시 입력하게 하는 것은 같은 일을 두 번 하는 셈이고, LLM 호출을 한 번 더 쓸 뿐 얻는 것이 없다. 그래서 불러오는 즉시 세션을 종료 상태로 두고, 3.2의 "대화 종료하기" 버튼이 곧바로 나타나게 한다. 결과 화면으로 넘어가는 시점은 여전히 사람이 그 버튼을 눌러 정한다.

진입점은 인터뷰 화면 안에 둔다. 시작 화면에는 두지 않는다. 참여자 흐름의 첫 화면이 개발용 요소로 오염되지 않게 하기 위해서다. 인터뷰 화면에서도 눈에 띄지 않는 작은 버튼으로 두어 참여자용 요소와 혼동되지 않게 한다.

대화 묶음집은 손으로 지어내지 않는다. 시뮬레이션 참여자를 세워 실제 Gemini 인터뷰를 완주시키고, 그때 나온 대화록과 증거를 그대로 저장한다. 지어낸 대화로 만든 결과는 실제 결과와 다르므로 검증에 쓸 수 없다. 서로 다른 유형이 나오도록 성향이 대비되는 묶음집을 여러 개 준비한다.

주입된 세션은 인터뷰를 실제로 진행한 세션과 구별되지 않는다. 따라서 이후의 종료, 유형 판정, 보고서 생성, 수정, 제출 경로를 특별 취급 없이 그대로 탄다.

개발자 모드는 `dev_mode` 환경변수가 켜져 있을 때만 활성화한다. 기본값은 꺼짐이고 Cloud Run 배포 시에는 켜지 않는다. 꺼져 있으면 위 두 경로를 등록하지 않으며 프론트의 진입 버튼도 나타나지 않는다.

## 7. 데이터 스키마

### 7.1 메모리 세션

FastAPI 프로세스의 딕셔너리에 다음을 보관한다.

- `session_id`
- 출생연도
- 연 단위 추정 나이(`2040 − 출생연도`)
- 원본 대화 메시지 배열
- 증거 로그
- 상태: 진행 중, 종료, 결과 생성, 제출됨
- 유형 결과
- 보고서 초안
- 수정 횟수
- 생성 시각

원본 대화록은 턴 배열이 아니라 메시지 배열이다. 각 메시지의 구조는 Firestore의 `raw_transcript`와 동일하다. 사용자 발화와 그에 대한 AI 응답은 같은 턴 번호를 공유한다.

서버 재시작 시 진행 중 세션이 소멸해도 허용한다. 제출 시에만 Firestore에 저장하고 메모리에서 제거한다.

### 7.2 증거

증거 한 건은 다음 필드로 구성한다.

- `axis`: `"EI"`, `"SN"`, `"TF"`, `"JP"` 중 하나
- `pole`: 해당 축의 극 글자
- `weight`: 1, 2, 3 중 하나
- `text`: 참여자 발화 원문
- `turn`: 턴 번호

### 7.3 유형 결과

`type_result`는 다음을 포함한다.

- `code`: 4글자 유형 코드
- `axes`: 네 축의 배열

각 축 객체는 다음을 포함한다.

- `axis`
- `letter`
- `strength`: 51~100 정수
- `scores`: 두 극의 가중치 합
- `evidence`: 해당 축의 증거 배열

모든 값은 증거 로그와 확정 합산 규칙으로 결정적으로 도출한다.

`scores`와 `evidence`는 서버 보관·저장용 완전본에 유지한다. 참여자에게 보내는 `/result` 응답에서는 두 필드를 제외하고 `axis`, `letter`, `strength`만 전달한다.

### 7.4 개인 보고서

개인 보고서는 다음 필드를 갖는다.

- `session_id`
- `self_info`
- `summary`
- `axis_reasons`
- `axis_demands`
- `meta`

`axis_reasons`는 네 축의 배열이며 각 항목은 `axis`, `letter`, `reason`을 갖는다. `reason`은 그 축을 왜 그 글자로 판정했는지를 설명하는 한두 문장이다. 참여자의 발화를 그대로 인용하지 않고, 참여자가 무엇을 중요하게 여겼기에 이 판정에 이르렀는지를 서술한다. 유형 결과와 마찬가지로 수정할 수 없으며 요구를 수정해도 재생성하지 않는다.

`self_info`는 다음을 포함한다.

- `nickname`
- `birth_year`
- `age_2040`
- `region`
- `dream_or_job`

`age_2040`은 연 단위 추정 나이(`2040 − 출생연도`)다. 출생연도는 파트 A에서 받고, 별명·지역·꿈 또는 직업은 대화록에서 LLM이 추출한다.

`summary`는 인터뷰 전체 요약의 문장 배열이며 수정할 수 없다.

`axis_demands`는 네 축의 배열이다. 각 축 객체는 다음을 포함한다.

- `axis`
- `letter`: 표시용 판정 글자
- `demands`

각 요구 항목은 다음을 포함한다.

- `id`: 예를 들어 `"EI-D1"`
- `title`
- `description`: 문장 단위 선택 UI를 위한 문장 배열
- `quotes`: `text`와 `turn`을 가진 근거 발화 배열

`quotes`는 참여자에게 보내는 `/result`와 `/revise` 응답에서 제외한다. 방금 자기가 한 말을 다시 읽는 것은 참여자에게 새로운 정보가 아니기 때문이다. 서버 세션과 Firestore 완전본에는 유지한다. 담당자가 제출본 상세에서 요구의 근거를 원문으로 확인할 수 있어야 하고, 분석 파이프라인이 대표 인용을 뽑는 재료이기도 하다.

수정 요청은 `axis`, `demand_id`, 문장 인덱스, 참여자 의견 텍스트로 구성된 배열이다. 선택 문장과 참여자 의견은 재생성 지시의 입력이 된다. LLM은 `axis_demands`의 일부가 아니라 4축 전체를 위와 동일한 스키마로 재생성하며, LLM 출력 자체에는 새 `axis_demands`만 담는다. 서버가 기존 `summary`, `axis_reasons`, `self_info`, 메타데이터를 복사해 전체 개인 보고서를 조립한다. `summary`와 별도 `type_result`는 변경하지 않는다.

`meta`는 다음을 포함한다.

- `turn_count`
- `revision_count`
- `created_at`

### 7.5 Firestore `submissions`

문서 ID는 UUID 형식의 `submission_id`다.

필드는 다음과 같다.

- `session_id`
- `submitted_at`
- `self_info`
- `raw_transcript`
- `evidence_log`
- `type_result`
- `report`
- `deidentified`

`raw_transcript`는 턴 배열이 아니라 메시지 배열이다. 각 메시지는 다음 필드를 갖는다.

- `turn`: 턴 번호
- `role`: `user` 또는 `assistant`
- `text`
- `timestamp`

한 사용자 메시지와 그에 대한 AI 메시지는 같은 `turn` 번호를 공유한다.

`deidentified`는 제출 응답 후 백그라운드에서 채우는 비식별 사본이다. 백그라운드 생성이 실패하거나 아직 끝나지 않아 값이 없으면 분석 실행 시 보충하며, 이미 있으면 재사용한다.

### 7.6 Firestore `analysis_runs`

문서 ID는 UUID 형식의 `run_id`다.

최상위 필드는 다음과 같다.

- `executed_at`
- `input_submission_ids`
- `axis_stats`
- `type_distribution`
- `axis_summaries`

`axis_stats`는 네 축의 배열이며 다음을 포함한다.

- `axis`
- 두 극의 인원수
- 극별 평균 강도
- 판정 근거가 된 제출본 목록

`type_distribution`은 유형 코드별 인원수다. 약 10명의 시범 규모에서는 참고용이고 축 단위 통계가 주력이다.

`axis_summaries`는 네 축의 배열이며 다음을 포함한다.

- `axis`
- 각 극 참여자의 요구 경향을 정리한 문장 배열
- 대표 인용 배열

대표 인용은 다음을 포함한다.

- `quote_id`
- 출처 `submission_id`
- 비식별 발화 텍스트

인용 ID 형식은 `{submission_id 앞 8자}-Q{순번}`이다. 예시는 `a1b2c3d4-Q2`다.

## 8. Gemini와 LLM 자산

### 8.1 모델

모든 LLM 작업에는 `gemini-3.5-flash`를 사용하되, 모델명을 코드에 하드코딩하지 않고 설정값으로 주입한다.

사용 대상은 다음과 같다.

- 인터뷰와 증거 태깅
- 개인 보고서 구조화
- 비식별화
- 축별 정성 요약과 대표 인용 선정

유형 합산과 축별 통계 집계는 LLM이 아니라 코드가 수행한다. 모델·프롬프트 버전 관리 기능은 만들지 않는다.

임베딩 모델은 없다. `gemini-embedding-001`도 사용하지 않는다.

### 8.2 프롬프트 파일

모든 LLM 자산은 `api/prompts/` 아래의 마크다운 파일로 두고 런타임에 로드한다. 파일은 정확히 다음 여섯 개다.

- `system.md`
  - 인천 시민의 미래 도시 상상을 돕는 인터뷰어 페르소나
  - 존댓말
  - 상상 보조
  - 금지사항과 가드레일
  - 악성 응답 처리
  - 4축 커버리지
  - 모든 어시스턴트 응답의 EVIDENCE 트레일러
  - 첫 인사의 빈 EVIDENCE 배열
  - 종료·중단 제어 토큰 규칙
  - `[운영 지시: ...]` 블록은 참여자 발화가 아니라 백엔드 운영 지시라는 규칙
  - 운영 지시 블록을 참여자 의견이나 증거로 해석하지 않는 규칙
  - 인터뷰어 이름은 임시로 "하늘"

- `rubric.md`
  - 도입: 라포 형성과 별명·거주지·꿈 수집, 연 단위 추정 나이 언급
  - 심화: 4축 주제 블록과 하루·동네를 구체화하는 꼬리질문
  - 정리: 요약, 확인, 마무리

- `axes.md`
  - 4축과 극의 정의
  - 축·극·강도별 채점 앵커
  - 루브릭과 증거 태깅이 공유하는 단일 원천

- `structuring.md`
  - 원본 대화록을 개인 보고서로 구조화
  - 인터뷰어 발화는 맥락으로만 사용하고 요구·인용은 참여자 발화에서만 추출
  - 자기보고 정보 추출
  - 축별 요구 생성
  - 수정 시 선택 문장과 의견을 바탕으로 `axis_demands` 4축 전체만 재생성

- `deidentify.md`
  - 이름과 연락처 블라인드

- `aggregate.md`
  - 축별·극별 요구 경향 요약
  - 대표 인용 선정

별도 `typing.md`는 만들지 않는다. 유형 채점 기준은 `axes.md`에 통합한다.

### 8.3 도시기본계획 지식 주입

참여자는 2040 인천 도시기본계획이 실현된 미래를 상상한다. 해당 계획이 채택된 역사적 배경은 구현에 반영하지 않는다.

선행 데이터 작업은 다음과 같다.

- 대용량 도시기본계획 PDF를 마크다운으로 전처리한다.
- 장별로 나누고 표·이미지를 정리한다.
- 결과는 `data/plan2040/`에 둔다.
- 사람이 직접 요약한 파일을 `data/plan2040_summary.md`에 둔다.

지식 주입은 두 방식으로 한다.

- 사람 작성 요약본은 시스템 프롬프트에 항상 포함한다.
- 전처리된 전체 마크다운은 Gemini API File Search Tool에 업로드한다.

File Search는 Google이 청킹, 임베딩, 저장, 검색을 관리하는 RAG 도구다. 별도 벡터 저장소를 만들지 않는다. 저장·질의 시 임베딩은 무료이고 최초 인덱싱만 과금되며 출처 인용이 자동으로 붙는다는 전제로 사용한다.

File Search 스토어는 GCP 프로젝트당 하나만 사용한다. Phase 1에서 업로드 스크립트가 스토어를 생성하고 전처리 문서를 업로드·인덱싱한다. 스토어 이름은 `api`의 환경변수로 주입하며 코드에 하드코딩하지 않는다.

매 인터뷰 턴의 `generateContent` 호출에 File Search를 부착해 현재 참여자 발화와 관련된 계획 내용을 검색하게 한다.

### 8.4 대화 맥락과 캐싱

인터뷰 도중과 종료 후 모두 대화록을 압축하지 않는다. 목표 15턴 동안 인터뷰 호출에는 전체 대화 기록을 전달하고, 종료 후 구조화·수정 호출에는 `turn`, `role`, `text`만 남긴 대화록을 전달한다.

프롬프트 순서는 다음 프리픽스 구조로 조립한다.

1. 시스템 프롬프트, 루브릭, 요약본으로 구성된 고정부
2. 누적 대화 기록
3. 현재 사용자 발화와 그 뒤에 덧붙인 `[운영 지시: ...]` 블록

Gemini implicit caching을 사용한다. 앞부분을 고정해 자동 캐시 할인을 유도한다. explicit 캐시 API는 Flash 요율에서 시간당 저장비가 절감분을 상쇄하므로 사용하지 않는다.

## 9. 데이터베이스, 배포, 운영

### 9.1 Firestore

Cloud Firestore Native 모드를 사용한다.

- 대화록과 보고서가 JSON 문서 구조에 적합하다.
- 시범 규모는 무료 티어로 충분하다고 본다.
- Google Cloud와 Gemini 생태계에 맞춘다.
- 로컬 개발도 동일 GCP 프로젝트의 Firestore를 사용한다.
- 에뮬레이터는 사용하지 않는다.
- 애플리케이션 수준의 추가 암호화를 하지 않는다.

### 9.2 Cloud Run

- 리전은 서울 `asia-northeast3`다.
- `web`과 `api` 두 서비스를 배포한다.
- 각 서비스는 Dockerfile을 가진다.
- 이미지는 Artifact Registry에 둔다.
- `api`는 프로세스 메모리 세션 때문에 `min=1`, `max=1`로 고정한다.
- 수평 확장은 하지 않는다.
- `min=1`로 콜드스타트를 줄여 시연 안정성을 확보한다.
- 시범 규모 약 10명에 충분한 구성으로 본다.
- Gemini 키, `gemini-3.5-flash` 모델명, File Search 스토어 이름 등은 환경변수로 주입한다.

`api`가 읽는 환경변수 이름은 다음과 같이 고정한다. 루트 `.env`에서 로드하고 Cloud Run에서는 서비스 환경변수로 주입한다. 값이 비어 있어도 기동은 되며, 해당 기능을 쓰는 시점에 필요하다.

- `gemini_api_key`: 필수
- `gemini_model`: 기본값 `gemini-3.5-flash`
- `file_search_store_name`: Phase 1의 업로드 스크립트가 스토어를 만든 뒤 주입
- `gcp_project_id`: Phase 4의 Firestore 연결에 사용
- `interview_target_turns`: 기본값 15
- `interview_wrapup_turn`: 기본값 12
- `axis_min_evidence`: 축이 충분히 탐색되었다고 볼 최소 증거 건수. 기본값 2
- `dev_mode`: 개발자용 대화 묶음집 경로 등록 여부. 기본값 `false`, Cloud Run에서는 활성화하지 않음

턴 상수를 코드가 아니라 환경변수로 두는 이유는 시범운영에서 인터뷰 길이를 재배포 없이 조정하기 위해서다.

`web`은 `NEXT_PUBLIC_API_BASE_URL`과 개발자 진입 UI를 빌드 시점에 제어하는 `NEXT_PUBLIC_DEV_MODE`를 읽는다. Cloud Run에서는 `NEXT_PUBLIC_DEV_MODE`를 활성화하지 않는다. Gemini 키와 프롬프트는 프론트에 두지 않는다.

### 9.3 공유 카드

프론트에서 유형 카드를 이미지로 변환한다. 이미지 변환 라이브러리는 `html-to-image`로 확정했다. 카드를 3배 해상도로 래스터화해 휴대폰에서 흐릿하지 않게 한다.

한글 서체를 애플리케이션에 번들하므로, 변환 시 그 서체가 이미지에 실제로 박혔는지 확인한 뒤 내보낸다. 서체가 인라인되지 않으면 시스템 서체로 조용히 대체되어 카드가 망가지므로, 그 경우에는 조용히 넘어가지 않고 그 공유·저장 시도를 실패로 끝낸다. 다만 공유·저장 실패는 결과 흐름 전체를 종료하지 않는다. 참여자는 결과 화면에 그대로 남아 다시 시도하거나 공유 없이 진행할 수 있다. 부수 기능의 실패로 제출 전 인터뷰 결과 전체가 사라지지 않게 하기 위함이다.

모바일에서는 Web Share API로 공유 시트를 열고, 파일 공유를 지원하지 않는 환경에서는 이미지를 내려받는다.

제출 이후에는 서버에서 유형 결과를 다시 조회하지 않는다. 프론트가 결과 단계에서 받아 보관 중인 `type_result` 상태로 공유 카드를 렌더링한다. `submit` 응답은 `submission_id`만 반환하며, 새로고침하거나 재접속하면 결과를 다시 볼 수 없다.

### 9.4 로깅

구조화 JSON 로그를 표준출력에 남기고 Cloud Run이 Cloud Logging으로 수집하게 한다.

로그 필드는 다음과 같다.

- 시각
- 레벨
- `session_id`
- 이벤트명
- 토큰 사용량

이벤트에는 세션 생성, 턴 진행, 종료, 이탈, 악성 중단, 결과 생성, 수정, 제출, 분석 실행 등이 포함된다.

대화 원문과 보고서 본문은 로그에 남기지 않는다. 원문은 제출 후 DB에만 존재한다.

### 9.5 테스트와 품질 도구

pytest로 다음 핵심 로직의 테스트를 기능 구현보다 먼저 작성한다.

- 세션 관리
- 제어 토큰과 트레일러 파싱
- 프롬프트 조립
- 스키마 검증
- 유형 가중치·반올림·동률·빈 축 처리
- 축별 집계 계산

테스트는 로컬 개발 자산으로만 유지하고 공개 저장소에는 포함하지 않는다. 저장소에는 시스템 구동에 필요한 코드만 둔다. 테스트를 작성하지 않는다는 뜻이 아니라 공개 범위에서 제외한다는 뜻이며, 위의 선행 작성 원칙은 그대로 지킨다.

Python은 ruff, 프론트는 ESLint와 Prettier를 사용한다.

### 9.6 에러 처리

존재하지 않는 세션은 404, 세션 상태에 맞지 않는 호출은 409, 그 외 서버 오류는 500으로 처리한다. 자동 재시도와 복구 로직은 만들지 않는다.

API 오류가 발생하면 고정 문구 "일시적인 오류가 발생했습니다"와 처음 화면으로 이동하는 버튼 하나만 있는 오류 화면을 표시하고 해당 흐름을 종료한다. 그 외 복구 안내나 복구 UI는 만들지 않는다. 시연 직전에 서비스 상태를 점검하는 것으로 대응한다.

## 10. 개인정보와 안전

수집 항목은 다음과 같다.

- 휴대폰 본인인증 통과 여부
- 출생연도
- 별명
- 거주 지역
- 꿈 또는 현재 직업
- 인터뷰 대화록
- 파생된 유형 결과와 보고서

휴대폰 신원은 중복참여 방지에만 사용하고 출생연도만 인터뷰 데이터로 전달한다. 다만 프로토타입의 목업 인증은 중복참여를 실제로 강제하지 않는다.

분석 전 Gemini가 이름과 연락처를 블라인드 처리한다. 담당자 종합 리포트와 대표 인용은 비식별본을 사용한다.

Firestore에는 애플리케이션 수준의 추가 암호화를 하지 않고 저장한다. 원본 접근은 담당자와 개발자로 한정하되, 구현 수단은 비공개 `/admin` 링크뿐이다. 로그인과 권한 시스템은 만들지 않는다.

진입 화면에는 수집 항목, 이용 목적, 보관 기간, 파기 시점을 고지한다. 실제 문구는 Phase 7에서 작성한다.

프롬프트 인젝션은 시스템 프롬프트 지시로만 방어한다. Gemini 안전필터의 오작동은 개발 단계에서 다루지 않는다.

- 도시기본계획 전처리 문서가 아직 없으므로 `data/plan2040/` 업로드와 File Search 스토어 생성은 실행하지 못했다. 코드 경로는 완성되어 있고, 문서가 채워지고 업로드 스크립트를 한 번 실행해 스토어 이름을 환경변수에 넣으면 코드 변경 없이 활성화된다. 그때까지 인터뷰는 요약본과 File Search 없이 동작한다.

시범 인터뷰에서 드러난 문제와 그 대응을 기록해 둔다. 이후 튜닝의 출발점이다.

- LLM이 4축을 한 번씩만 찍고 8턴 만에 종료했다. 축당 증거가 한두 건에 그쳐 강도가 100%로 계산되는 문제가 있었다. 커버 기준을 증거 2건으로 올리고 12턴 전 마무리 금지 안내를 넣어, 13턴·증거 16건·축당 3건 이상으로 개선했다.
- LLM이 축의 두 극을 그대로 선택지로 내미는 이지선다 질문을 했다. 5.3의 검사형 질문 금지 위반이다. 원인은 루브릭의 주제를 "A인지 B인지" 형태로 적어 둔 데 있었다. 루브릭의 모든 주제를 열린 질문으로 다시 쓰고 시스템 프롬프트에 이지선다 금지를 명시해 해소했다.
- 한 발화에 여러 축이 섞여 있을 때 한 축만 태깅하고 나머지를 놓쳤다. 축별 점수가 0이 되어 유형이 왜곡된다. 발화 하나를 네 축 모두에 대해 훑으라는 규칙과, 중복 방지 규칙은 같은 축 안에서만 적용된다는 단서를 `axes.md`에 넣어 해소했다.
- 인터뷰어의 응답이 길고 감탄과 칭찬이 과했으며 마크다운 서식을 썼다. 세 문장 제한, 과장 금지, 평문 규칙을 시스템 프롬프트에 넣었다. 완전히 해소되지는 않았으므로 시범 전 추가 튜닝 대상이다. 이후 리뷰 라운드(R9)에서 세 문장 경성 상한은 기본 두세 문장, 필요 시 다섯 문장까지 허용하는 연성 규칙으로 완화했다.

검증: 사용자가 화면에서 인터뷰를 처음부터 끝까지 완주하고, 오류 시 지정된 단일 오류 화면으로 흐름이 종료된다.

완료 시점의 상태는 다음과 같다.

- 2.4의 디자인 계약을 구현했다. 인천 3색을 shadcn 시맨틱 토큰에 직접 대응시켜 버튼과 강조가 자동으로 인천 블루가 되게 했고, 파괴적 행동에도 빨강 대신 인천 그레이를 쓴다. Pretendard Variable을 번들해 외부 폰트 CDN에 의존하지 않는다.
- 데스크톱에서는 모바일 폭을 유지한 채 기기 형태로 가운데에 배치하고, 실제 휴대폰에서는 화면을 가득 채운다.
- 타자기 효과는 도착한 문자만 큐에 넣고 일정한 속도로 한 글자씩 공개한다. 스트림보다 앞서 나가지 않고, 스트림이 끝나면 남은 글자를 반드시 모두 공개한다. 밀린 글자가 많으면 글자 간 간격만 줄여 따라잡는다.
- 실제 브라우저에서 `시작 → 첫 인사 → 대화 → 그만두기 확인 → 세션 폐기 → 시작 화면 복귀`와 API 장애 시의 단일 오류 화면을 모두 확인했다.
- 결과 화면은 Phase 3의 몫이므로, 인터뷰가 정상 종료되면 완료 안내만 표시하고 멈춘다.

### Phase 3: 유형 판정, 구조화, 확인과 제출 (완료 — 핵심 MVP 도달)

백엔드:

- 유형 합산 코드
- 가중치·반올림·동률·빈 축 규칙의 선행 pytest
- `structuring.md`
- 개인 보고서 스키마
- 수정 시 `axis_demands` 4축 전체 재생성
- `summary`와 `type_result` 불변 처리

프론트:

- 3.3의 세 단계로 나뉜 결과 화면
- 4글자 유형과 별명
- 축별 강도 바
- 축별 판정 이유
- 축별 요구 보고서
- 문장 선택과 의견 입력
- 재생성 표시
- 제출 버튼
- 결과 단계의 `type_result` 상태 유지
- 제출 전후 모두에서 이미지 저장·공유
- 새로고침 시 결과를 다시 볼 수 없다는 안내

16유형 콘텐츠 초안을 `web/city_types.json`에 `code`, `nickname`, `description` 스키마로 작성하고 팀 검수를 병행한다.

검증: 한 사용자가 `인터뷰 → 유형·보고서 확인 → 4축 요구 재생성 → 제출 → 공유 카드`를 완주하며, 제출 응답이 `submission_id`만 반환해도 기존 `type_result` 상태로 공유 카드가 렌더링된다.

이 단계 완료를 실시간 구간 완성과 핵심 인터뷰 MVP 도달로 본다.

완료 시점의 상태는 다음과 같다.

- 유형 합산은 LLM 없이 코드가 결정적으로 수행한다. 반올림은 파이썬 기본 `round`의 은행가 반올림을 피해 `floor(x + 0.5)`로 구현했고, 두 극의 표시 백분율은 승자만 반올림한 뒤 패자를 `100 − 승자`로 유도해 합이 항상 정확히 100이 되게 했다.
- 결과 화면은 3.3의 세 단계로 나뉜다. 유형 코드, 별명, 한 줄 설명, 축별 강도 바, 축별 판정 이유, 축별 요구 보고서를 보여 준다. 유형·강도·판정 이유·요약은 고정이고 수정할 수 없다.
- 요구 수정은 문장 선택과 의견 입력으로만 이루어지며, LLM이 4축 전체를 재생성한다. `summary`와 `type_result`의 불변은 프롬프트 지시가 아니라 코드로 강제한다. 재생성 결과에서 `axis_demands`만 취하고 나머지는 기존 보고서에서 복사한다.
- 제출은 `submission_id`만 반환하고 세션을 메모리에서 제거한다. Firestore 저장은 Phase 4에서 이 자리에 붙인다.
- 공유 카드는 제출 후 서버를 다시 조회하지 않고 프론트가 들고 있던 `type_result`로 렌더링한다.
- 실제 브라우저에서 `개발자 모드로 대화 불러오기 → 마무리 발화 → 종료 → 유형 카드 → 요구 수정 → 제출 → 공유 카드 PNG 생성`을 완주해 확인했다. 번들한 한글 서체가 이미지에 실제로 박히는 것도 확인했다.

시범 전에 살펴야 할 사항을 하나 기록해 둔다. 어떤 축에서 참여자가 반대 극을 한 번도 말하지 않으면 5.3의 공식상 강도가 100%로 계산된다. 데이터로는 옳지만 화면에 100%와 0%가 나란히 표시되면 단정적으로 읽힐 수 있다. 실제 참여자는 대개 양극을 섞어 말하므로 시범운영에서 실제 분포를 보고 표현 방식을 다시 판단한다.

## 13. 검증과 리스크 관리

시범운영에서는 다음을 측정한다.

- 질문 구성이 자연스러운가
- 막연한 상상이 구체적인 정책 수요로 발전하는가
- 4축을 모두 탐색하는가
- 빈 축 발생률
- 참여자가 유형 판정에 납득하는가
- 요약과 축별 분석이 정확한가
- 비식별화가 완전한가
- 인터뷰 완주율
- 확인·수정 후 제출 비율
- 공유 버튼 사용 여부

효과 검증은 청년 참여자를 A·B 그룹으로 나눠 진행한다.

- A그룹: 일반 설문조사
- B그룹: 유스플랜AI

두 결과를 요약해 전문가가 비교 평가한다.

주요 리스크와 대응은 다음과 같다.

- 인터뷰 환각·유도질문·피상성: 프롬프트 계층, 가드레일, 시범 전 반복 검증
- 빈 축·납득하기 어려운 유형: 루브릭 커버리지, 미커버 축 힌트, 증거 인용, 채점 앵커 튜닝
- 시연 장애: `api min=1`, 사전 상태 점검, Phase 8 통주 리허설
- 개인정보: 신원 분리, 출생연도만 전달, 비식별화, 로그 원문 배제
- 참여자 이탈: 앱 설치 없는 모바일 진입, 유형 보상, 확인·수정 경험
- 일정 압박: 좁은 MVP, 실시간·배치 분리, 프롬프트 파일 후행 튜닝

## 14. 미결 사항

- 4축 v1은 그대로 구현하되 시범 전 `axes.md`의 표현과 채점 앵커를 튜닝한다.
- 개발자 모드 양가형 묶음집(`api/fixtures/mina.json`) 실험에서, 양극이 섞인 인물인데도 T/F 축은 T 증거가 하나도 잡히지 않고 F로 전면 쏠렸다(E/I·S/N·J/P는 양극 증거가 섞여 강도 57~69%의 중간대로 형성). 태깅이 T/F 축에서 T 지향(효율·일자리·경쟁력)을 놓치는 블라인드스팟이 확인됐으므로, 다음 라운드에서 `axes.md`의 T 채점 앵커 보강을 검토한다. 반대로 나머지 세 축의 결과는 기존 두 묶음집(junwoo·ssu)의 전면 100% 쏠림이 대체로 대비형 인물 설계 탓임을 보여 준다.
- 16유형 별명·설명은 Claude 초안 후 팀이 확정한다.
- 인터뷰어 이름은 임시로 "하늘"을 사용하며 추후 마크다운 파일에서 교체한다.
- 특정 회사·세부 지역 같은 준식별자 처리 기준은 베타 테스트에서 정한다.
- 개인정보 수집·이용 고지 문구는 시작 화면에 초안을 넣었으며 2026-07-17 팀 회의에서 확정한다.
- 어떤 축의 반대 극 증거가 하나도 없을 때 강도가 100%로 표시되는 문제는 시범운영의 실제 분포를 보고 표현 방식을 다시 판단한다.
- 담당자용 축·유형 해설은 `web/` 정적 자산으로 두며 구체적인 파일 위치는 Phase 6 착수 시 결정한다.
- 참여자가 브라우저를 닫으면 이탈 세션이 프로세스 메모리에 남는다(DELETE 미수신). 시범 규모(약 10명)에서는 실害가 없으므로 TTL 스윕은 만들지 않고, Cloud Run 장기 운영 시점에 재평가한다.

## 15. 현재 진행 상황

새 세션이 이 절만 읽고도 이어받을 수 있도록 적는다. 기준일은 2026-07-15이다.

### 어디까지 됐나

**Phase 0부터 3까지 완료했다.** 핵심 인터뷰 MVP가 실제로 동작한다. 참여자가 인터뷰를 완주하면 코드가 유형을 판정하고, LLM이 개인 보고서를 만들고, 참여자가 요구를 수정하고 제출한 뒤 유형 카드를 이미지로 저장하거나 공유하는 흐름이 실제 Gemini로 끝까지 돌아간다.

Phase 3 이후 실사용 검토에서 나온 개선도 모두 반영했다. 각각의 계약은 해당 절에 적혀 있다.

- 인터뷰가 끝나도 화면이 자동으로 넘어가지 않고 "대화 종료하기" 버튼을 기다린다 (3.2)
- 결과 화면을 세 단계로 나눴다. 한 화면에 몰아넣으니 스크롤이 지나치게 길었다 (3.3)
- 판정 근거로 참여자 발화를 다시 인용하지 않고, 축마다 왜 그 글자인지를 설명한다 (3.3, 7.4)
- 보고서 생성은 압축 없이 원본 대화록을 한 번에 구조화한다 (3.3, 8.2)
- 참여자 응답에서는 서버 보관용 증거·점수·인용 원문을 제외한다 (6.1, 7.3, 7.4)
- 시작 화면에 서비스 설명과 본인인증 버튼을 두었다 (2.4)
- 개발자 모드는 대화를 불러오는 즉시 세션을 종료 상태로 둔다 (6.4)

### 시작하는 법

개발 서버는 두 개를 띄운다.

```
cd api && uv run uvicorn app.main:app --port 8000
cd web && pnpm dev
```

브라우저에서 `http://localhost:3000`으로 접속한다. UI를 확인할 때는 인터뷰를 직접 타이핑하지 말고 6.4의 개발자 모드를 쓴다. 검증 방법은 `CLAUDE.md`의 "UI 검증 방법"에 적혀 있다. `cd web && pnpm e2e`가 45초 만에 전 흐름을 검사하고 스크린샷을 남긴다.

### 다음에 할 일

Phase 4부터 8까지 남았다. 순서대로 하지 않아도 되는 것이 있다.

- **Phase 4 (Firestore)**: GCP 프로젝트가 있어야 시작할 수 있다. 지금 제출은 `submission_id`만 반환하고 메모리 세션을 지운다. 저장 코드가 들어갈 자리는 비워 두었다.
- **Phase 5 (분석 파이프라인)**: 제출본이 있어야 하므로 Phase 4에 묶여 있다.
- **Phase 6 (담당자 웹)**: GCP 없이도 진행할 수 있다.
- **Phase 7 (진입 게이트)**: GCP 없이도 진행할 수 있다.
- **Phase 8 (배포)**: GCP가 필요하다.

### 막혀 있는 것

- **GCP 프로젝트**가 없다. Phase 4와 8에 필요하다. `gcloud` CLI도 설치되어 있지 않다.
- **도시기본계획 전처리 문서**가 없다. `data/plan2040/`는 비어 있고 `data/plan2040_summary.md`는 만들지 않았다. 코드 경로는 완성되어 있으므로, 문서를 채우고 `api/scripts/upload_plan.py`를 한 번 실행해 스토어 이름을 `.env`에 넣으면 코드 변경 없이 활성화된다. 그때까지 인터뷰는 이 지식 없이 동작한다.
- **README**는 아직 한 줄짜리다. 마지막에 작성하기로 했다.
